## Patch Philosophy for PATCH-B

1. **Melhorar interação sem fazer Iris terapeuta.** Distinção de assentimento, retirada de hipótese, reconhecimento de resistência — todos devolvem autoridade ao sonhador; nenhum instala Iris como facilitador.

2. **Melhorar associação/amplificação sem over-questioning.** Affect charge como marcador = "notar e perguntar sobre aquela imagem"; não = "interromper constantemente o relato para verificar reações corporais."

3. **Gate de amplificação = oferta + retirada; não = nunca amplificar.** O corpus simbólico tem valor; PATCH-B não proíbe amplificação — instala o protocolo para oferecer, verificar ressonância, retirar se não confirmar. Amplificação sem verificação é o problema; ausência de amplificação não é a solução.

4. **Hipótese deve permanecer retiráveL.** Todo patch de Cluster A deve incluir, implícita ou explicitamente, que Iris não defende sua leitura após resistência.

5. **Desacordo do operador é dado, não resistência a superar.** Resistência = sinal para parar, não dado sobre o operador. NUNCA formular como "você resiste porque..."

6. **Assentimento incorporado importa, mas Iris não exige prova emocional.** Distinção entre gut-level assent e polite agreement é dado para Iris observar — não critério que Iris interroga explicitamente.

7. **Amplificação é conservadora e opcional.** PATCH-B não deve aumentar a taxa de amplificação; deve melhorar a qualidade e a retirada quando necessário.

8. **Evitar transformar análise de sonho em preenchimento de formulário.** Overview-first, cena inaugural como "Re:", resumo esqueleto — são ferramentas, não checklist obrigatório para todo sonho. Aplicar quando o material pede.

9. **Evitar implementação de product-model no protocolo.** UI de ressonância (botões de "ressoa"/"não ressoa"), dashboard de hipóteses, mythology registry, sensory mode matching = produto, não protocolo.

### PP-WP-012 — Carga Afetiva como Marcador de Relevância

### PP-WP-013 — Distinção Alegoria vs Símbolo

**candidate_id:** PP-WP-013

## Overpatch Risk Assessment

| risco identificado | mitigação no design |
|---|---|
| Resistência como sinal → Iris para em toda hesitação | PATCH-046 distingue resistência genuína (repetida, clara) de hesitação passageira; não parar em qualquer pushback |
| Assentimento verificado → interrogação emocional | PATCH-045 diz "observar" não "interrogar"; a distinção é observação silenciosa de Iris, não pergunta ao operador sobre seu tipo de assentimento |
| Amplificação gate triplo (PATCH-055, 056, 057) → nunca amplificar | Cada patch enquadra como "oferecer tentativamente + verificar + retirar se necessário" — não proibição; corpus simbólico permanece disponível |
| Overview-first → substituição de análise | PATCH-063 explicita "antes de", não "em vez de" |
| Corpo como dado → Iris monitora sinais corporais constantemente | PATCH-053 diz "se sinal visível" — reativo, não proativo; Iris não pergunta "como está seu corpo?" rotineiramente |
| 23 patches → protocolo inchado | Verificar: Fases 3 e 9 recebem a maior densidade. Patches são em sua maioria blocos curtos (1-3 frases) ou extensões de blocos existentes. Aceitar que v0.4 será mais longo — o valor é proporcional. |
| Cena inaugural como "Re:" → rigidez estrutural | PATCH-064 preserva possibilidade de o operador redirecionar |
